API 一旦发布，就有了消费者。如何在不破坏现有客户端的前提下演进 API，是每个后端团队都必须面对的问题。本文将深入对比三种主流的 API 版本策略。

为什么需要 API 版本控制

当 API 需要进行破坏性变更时（删除字段、修改数据结构、更改业务规则），如果没有版本控制，所有客户端会同时受到影响。版本控制让新旧 API 共存，给客户端充足的迁移时间。

URL 路径版本（Path Versioning）

这是最常见也最直观的方式，将版本号嵌入 URL 路径中。

  • 格式 —— /api/v1/users、/api/v2/users
  • 优点 —— 直观、易于理解、便于测试和调试
  • 缺点 —— URL 不够"纯净"，版本号并非资源的一部分
  • 适用场景 —— 面向外部开发者的公开 API

请求头版本（Header Versioning）

通过自定义请求头来指定 API 版本，保持 URL 的简洁。

  • 格式 —— Accept: application/vnd.cyclemind.v2+json
  • 优点 —— URL 保持整洁，符合 REST 纯粹主义
  • 缺点 —— 不够直观，浏览器直接访问不方便，文档编写复杂
  • 适用场景 —— 内部微服务间通信

查询参数版本（Query Param Versioning）

通过查询参数传递版本号，是一种折中方案。

  • 格式 —— /api/users?version=2
  • 优点 —— 实现简单，可以设置默认版本
  • 缺点 —— 容易被忽略，缓存策略复杂
  • 适用场景 —— 快速迭代的内部项目

三种策略对比

没有绝对最优的方案，选择取决于你的具体场景。

对于大多数团队，URL 路径版本是最安全的选择——简单、明确、不容易出错。

在 CycleMind 生成的 API 设计文档中，默认采用 URL 路径版本策略，但你可以通过配置切换到其他方案。

版本迁移指南

发布新版本 API 时，必须为客户端提供清晰的迁移指南。

  1. 提前公告 —— 至少提前两个迭代周期通知
  2. 提供迁移文档 —— 详细列出所有破坏性变更
  3. 并行运行 —— 新旧版本至少并行运行 3-6 个月
  4. 监控旧版本流量 —— 确认所有客户端已迁移后再下线
  5. 设置日落日期 —— 明确旧版本的停止服务日期

向后兼容性

并非所有变更都需要新版本。以下类型的变更是向后兼容的：

  • 新增可选字段 —— 不影响现有客户端
  • 新增端点 —— 现有端点不受影响
  • 新增枚举值 —— 客户端应设计为能容忍未知值

需要新版本的变更包括：删除字段、修改字段类型、更改必填/选填属性、修改业务语义等。

总结

API 版本控制是 API 生命周期管理的关键环节。选择合适的策略、制定清晰的迁移计划、保持向后兼容性，这些都是构建可信赖 API 的基础。CycleMind 在生成 API 设计文档时会自动考虑版本策略，帮助你从第一天就做好准备。
