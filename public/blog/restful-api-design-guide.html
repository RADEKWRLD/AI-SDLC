RESTful API 是现代 Web 应用的基石。一套优秀的 API 设计规范不仅能提升开发效率，更能让前后端协作如丝般顺滑。本文将从 URL 命名到错误处理，系统性地梳理 RESTful API 设计的核心原则。

URL 命名规范

资源命名是 API 设计的第一步，也是最容易被忽视的一步。

  • 使用名词复数 —— /users、/projects，而非 /getUser
  • 层级关系用嵌套表达 —— /projects/42/documents 表示项目下的文档
  • 避免动词 —— 动作由 HTTP 方法承载，URL 只描述资源
  • 使用 kebab-case —— /design-documents 而非 /designDocuments

HTTP 方法语义

每个 HTTP 方法都有明确的语义约定，不应混淆使用。

  • GET —— 读取资源，必须是幂等且安全的
  • POST —— 创建资源，返回 201 Created
  • PUT —— 全量替换资源，幂等操作
  • PATCH —— 部分更新资源
  • DELETE —— 删除资源，返回 204 No Content

状态码的正确使用

状态码是 API 与调用方之间的"语言"，精准的状态码能减少大量沟通成本。

  • 200 OK —— 请求成功
  • 201 Created —— 资源创建成功
  • 400 Bad Request —— 请求参数有误
  • 401 Unauthorized —— 未认证
  • 403 Forbidden —— 无权限
  • 404 Not Found —— 资源不存在
  • 422 Unprocessable Entity —— 参数格式正确但语义有误
  • 500 Internal Server Error —— 服务器内部错误

分页与过滤

当资源集合较大时，分页是必须的。CycleMind 生成的 API 设计文档会自动包含分页方案。

  • 偏移分页 —— ?page=2&limit=20，简单直观但大数据量性能差
  • 游标分页 —— ?cursor=abc123&limit=20，适合实时数据流
  • 过滤参数 —— ?status=active&sort=-created_at

错误处理

统一的错误响应格式能极大简化客户端的错误处理逻辑。

好的错误信息应该告诉开发者：发生了什么、为什么发生、以及如何修复。

建议的错误响应结构包含 code（机器可读错误码）、message（人类可读描述）和 details（字段级错误详情）。

HATEOAS 与内容协商

HATEOAS（超媒体作为应用状态引擎）让 API 具备自描述能力，客户端可以通过返回的链接发现可用操作。

内容协商通过 Accept 和 Content-Type 头实现，允许同一端点返回 JSON、XML 等不同格式。在实践中，大多数现代 API 默认使用 JSON，但保留内容协商能力是良好的设计习惯。

总结

RESTful API 设计没有绝对的"标准答案"，但遵循这些原则能让你的 API 更加一致、可预测、易于使用。在 CycleMind 中，AI 会根据你的需求描述自动生成符合这些规范的 API 设计文档，让你专注于业务逻辑本身。
